## Этап 0: Ollama + FastAPI на GPU-сервере

### Промпт для AI-агента


## Этап 3: 1С HTTP-сервис (расширение)

### Кто выполняет

**Человек** в конфигураторе 1С. AI-агент не нужен.

Можно добавить HTTP-сервис прямо в основную конфигурацию (если база уже
опубликована — заработает сразу после обновления БД). Расширение — если нет
доступа к основной конфигурации.

### Пошаговая инструкция

```
3.1  Создать расширение
     Конфигуратор → Конфигурация → Расширения конфигурации → Добавить
     Имя: AIAssistant, Префикс: aihlp_, Назначение: Дополнение

3.2  Добавить HTTP-сервис QueryService
     В расширении → Добавить → HTTP-сервисы → HTTP-сервис
     ┌─────────────┬────────────────────────────────────────┐
     │ Корневой URL │ ai                                     │
     │ Шаблон URL   │ /query                                 │
     │ Метод        │ POST → обработчик ОбработкаВызова      │
     └─────────────┴────────────────────────────────────────┘
     ВАЖНО: Корневой URL = "ai" (без слэшей, 1С не допускает "/" в этом поле).
     Итоговый путь: http://<server>/<base>/hs/ai/query
     Код модуля: скопировать из extension/README.md секция 2.3

3.3  Добавить роль ТолькоЧтениеВитрин
     В расширении → Добавить → Роли → Роль
     Права:
     [x] Чтение — витринные регистры накопления (из вводных)
     [x] Использование — HTTP-сервис QueryService
     [ ] Всё остальное — снять (запись, изменение, удаление, добавление)

3.4  Создать пользователя ai_assistant
     Администрирование → Пользователи → Добавить
     Роль: ТолькоЧтениеВитрин
     Аутентификация: базовая (HTTP Basic Auth)
     Сохранить пароль в .env на GPU-сервере:
       ONEC_USER=ai_assistant
       ONEC_PASSWORD=<пароль>
       ONEC_BASE_URL=http://<1c-server>/<base>/hs/ai

3.5  Публикация базы через Apache (если ещё не опубликована)
     Установить модуль: sudo apt install 1c-enterprise-*-ws
     Опубликовать: Администрирование → Публикация на веб-сервере
     В default.vrd добавить:
       <httpServices publishByDefault="true"/>
     Перезапустить: sudo systemctl restart apache2

3.6  Проверка
     curl -u ai_assistant:<пароль> -X POST \
       -H "Content-Type: application/json" \
       http://<1c-server>/<base>/hs/ai/query \
       -d '{"query":"ВЫБРАТЬ ПЕРВЫЕ 1 1 КАК Тест","params":{}}'
     Ожидание: {"success":true,"data":[{"Тест":1}],...}
```

### Промпт для AI-агента

```

Шаги:
```

```
     Вызвать LLM (GPU 0) с промптом prompts/router.txt + message.
     Парсить ответ: взять первое слово, привести к lower.

   - init_history(db_path) — подключение к history.db
   - create_session(user_id=None) → session_id
   - save_message(session_id, role, content, intent=None,
     register=None, query_text=None, latency_ms=None)

     Input:  {message: str, dashboard_context?: {url, title}, session_id?: str}

     Логика:
     a) Если нет session_id — create_session()

   - test_knowledge_intent — "как считается маржа" → "knowledge"

5. Интеграционный тест tests/test_chat_e2e.py:
   - test_data_flow_e2e — POST /chat с data-вопросом → ответ с цифрами
```

## Этап 7: Тюнинг и production hardening

### Промпт для AI-агента

```
Контекст: проект 1c-analytycs-ai-help. MVP работает.
Результаты тестирования 10 вопросов: {ВСТАВЬ РЕЗУЛЬТАТЫ ИЗ ЭТАПА 5}

Задача: улучшить качество ответов и подготовить к production.

Шаги:
1. Тюнинг промптов:
   - Для каждого некорректного ответа из тестирования —
     определи причину (router ошибся / wrong register / bad query / bad format)
   - Обнови соответствующий промпт в prompts/ с few-shot примерами
   - Добавь проблемные кейсы как примеры в промпты

2. Добавь шаблоны для вопросов, которые LLM генерировал плохо:
   - Добавь новые паттерны в query_templates.py

3. Логирование:
   - Добавь structured logging (json) в каждый компонент
   - Логируй: вопрос, intent, register, query, latency, errors
   - Файл: /var/log/ai-assistant/app.log

4. Мониторинг:
   - GET /metrics → prometheus-совместимые метрики:
     requests_total, request_duration_seconds, llm_calls_total,
     cache_hits_total, errors_total (by type)

5. Graceful error handling:
   - 1С недоступна → "Сервер данных временно недоступен, попробуйте позже"
   - Wiki недоступна → "База знаний временно недоступна"
   - LLM timeout → "Обработка заняла слишком много времени, попробуйте упростить вопрос"
   - Невалидный запрос после LLM → "Не удалось сформировать запрос, попробуйте переформулировать"

6. systemd юнит для FastAPI:
   - ai-assistant.service
   - auto-restart on failure
   - after ollama services
```
