# Smart 1C Backend: надёжный tool calling + сборка запросов на стороне 1С

**Дата:** 2026-04-10  
**Статус:** утверждён

## Проблема

1. Tool calling через Gemma 4 E2B (5.1B) нужно сделать максимально надёжным для продакшена
2. Текущие 4 шаблона запросов не покрывают потребности: нет сравнений, вычисляемых полей, произвольных фильтров
3. Модель 5B не способна самостоятельно генерировать корректный 1С-синтаксис
4. Приоритет: лучше не ответить, чем ответить неправильно

## Решение

Перенос логики сборки 1С-запроса из Python (`query_builder.py`) в HTTP-сервис 1С. Модель по-прежнему выбирает инструмент и заполняет JSON-параметры (зона комфорта 5B-модели), а 1С собирает и выполняет нативный запрос.

## Архитектура

### Текущая схема
```
User → Gemma (tool call) → JSON params → Python query_builder → текст запроса → 1С HTTP (execute) → JSON data
```

### Новая схема
```
User → Gemma (tool call) → JSON params → Python (валидация + нормализация) → 1С HTTP (build + execute) → JSON data
```

### Ключевой принцип

Текст 1С-запроса нигде не передаётся по сети. JSON-параметры — единственный контракт между Python и 1С. Это обеспечивает безопасность (нет инъекций) и надёжность (запрос всегда корректен синтаксически).

## Набор инструментов (7 штук)

### Существующие

| # | Tool | Когда | Пример вопроса |
|---|------|-------|----------------|
| 1 | `aggregate` | Одно число за период | "Какая выручка за март 2025?" |
| 2 | `group_by` | Разбивка по измерению | "Выручка по ДЗО за март" |
| 3 | `top_n` | Ранжирование | "Топ-5 ДЗО по выручке" |
| 4 | `time_series` | Динамика по месяцам | "Тренд EBITDA помесячно" |

### Новые

| # | Tool | Когда | Пример вопроса |
|---|------|-------|----------------|
| 5 | `compare` | Сравнение двух значений одного измерения | "Сравни выручку факт vs план за март 2025" |
| 6 | `ratio` | Вычисляемое отношение двух показателей | "Рентабельность = маржа / выручка за март" |
| 7 | `filtered` | Условие на значение агрегата | "ДЗО где выручка > 100 млн за март" |

### Контракты новых инструментов

**`compare`:**
```json
{
  "tool": "compare",
  "resource": "Сумма",
  "compare_by": "scenario",
  "values": ["Факт", "План"],
  "year": 2025, "month": 3
}
```

**`ratio`:**
```json
{
  "tool": "ratio",
  "numerator": "Маржа",
  "denominator": "Выручка",
  "year": 2025, "month": 3
}
```
`numerator`/`denominator` — значения измерения "Показатель" (например "Маржа", "Выручка"). Python нормализует их в `params.values: ["Маржа", "Выручка"]` и передаёт в 1С. 1С делает один запрос с `ГДЕ Показатель В (&Значения)` + `СГРУППИРОВАТЬ ПО Показатель`, затем делит первое на второе. Порядок: `numerator / denominator`.

**`filtered`:**
```json
{
  "tool": "filtered",
  "resource": "Сумма",
  "group_by": "company",
  "condition": {"operator": ">", "value": 100000000},
  "year": 2025, "month": 3
}
```
`operator` — enum: `>`, `<`, `>=`, `<=`, `=`.

### Риски для Gemma 5B

- `compare` — низкий риск (простой enum)
- `ratio` — средний риск (разложение "рентабельность" на числитель/знаменатель, помогут примеры в description)
- `filtered` — средний риск (парсинг "больше 100 млн" → `{">", 100000000}`, возможен fallback на переспрос)

## Контракт Python ↔ 1С HTTP-сервис

### Эндпоинт

**POST** `/analytics/execute`

### Запрос
```json
{
  "register": "РегистрНакопления.Бюджетирование.Обороты",
  "tool": "compare",
  "params": {
    "resource": "Сумма",
    "compare_by": "Сценарий",
    "values": ["Факт", "План"],
    "filters": {
      "КонтурПоказателя": "Общий",
      "Показатель": "Выручка"
    },
    "period": {
      "year": 2025,
      "month": 3
    }
  }
}
```

Python нормализует латинские ключи в кириллицу перед отправкой. 1С получает родные имена измерений.

### Успешный ответ
```json
{
  "success": true,
  "data": [
    {"Сценарий": "Факт", "Значение": 150000000},
    {"Сценарий": "План", "Значение": 200000000}
  ],
  "computed": {
    "diff": -50000000,
    "percent": -25.0
  },
  "query_text": "ВЫБРАТЬ ..."
}
```

`query_text` — опционально, для отладки.

### Ответ с ошибкой
```json
{
  "success": false,
  "error_type": "invalid_params",
  "error_message": "Неизвестное значение измерения 'Сценарий': 'Прогноз'. Допустимые: Факт, План, Бюджет",
  "allowed_values": ["Факт", "План", "Бюджет"]
}
```

### Типы ошибок

| error_type | Когда | Действие Python |
|---|---|---|
| `invalid_params` | Неизвестное значение измерения | Показать allowed values, переспрос |
| `missing_params` | Не хватает обязательного параметра | Переспросить конкретный параметр |
| `no_data` | Запрос выполнен, результат пустой | "Данные за указанный период не найдены" |
| `execution_error` | Ошибка выполнения запроса в 1С | Логировать, пользователю — "не удалось выполнить запрос" |

## Надёжность tool calling

### Сохраняем из текущей калибровки (issue #1)

- Латинские имена параметров (`metric`, `company`)
- `tool_choice: "required"`
- Descriptions на английском, enum values на русском
- Не перегружать `required` в JSON Schema

### Добавляем

**1. Fallback при отсутствии tool call:**
```
Response без tool_calls → retry 1 раз с усиленным промптом
Retry без tool_calls → "Не удалось обработать вопрос. Попробуйте переформулировать"
```

**2. Валидация параметров в Python (до сетевого вызова):**
- `resource` — есть в metadata
- `year` — 2020–2030
- `month` — 1–12
- `operator` (для `filtered`) — в enum
- `values` (для `compare`) — массив из 2 элементов

**3. Structured error → переспрос пользователя (не retry через модель)**

**4. Расширенный калибровочный набор (20+ кейсов):**

| # | Вопрос | Ожидание |
|---|--------|----------|
| 12 | Сравни факт и план по выручке за март 2025 | compare |
| 13 | Факт vs бюджет EBITDA за январь 2025 | compare |
| 14 | Рентабельность за март 2025 | ratio |
| 15 | Маржа к выручке за Q1 | ratio |
| 16 | ДЗО где выручка больше 100 млн за март | filtered |
| 17 | Показатели с суммой меньше 10 млн | filtered |
| 18 | Какая выручка (без периода) | missing_params → переспрос |
| 19 | Покажи данные (слишком общий) | переспрос |
| 20 | аыфвоаыф (мусор) | переспрос |

**5. Логирование:** каждый tool call — вопрос → tool → параметры → результат/ошибка.

## Изменения в Python-слое

### Удаляется
- `query_builder.py` — логика в 1С
- `query_validator.py` — текст запроса больше не передаётся

### Меняется
- `tool_defs.py` — 7 инструментов вместо 4
- `tool_caller.py` — retry-логика, нормализация новых tools
- `main.py` — упрощённый flow: tool_caller → param_validator → onec_client.execute_tool

### Добавляется
- `param_validator.py` — быстрая валидация JSON-параметров
- `onec_client.py: execute_tool()` — POST на `/analytics/execute`

### Flow после изменений
```python
tool_result = await tool_caller.call(question, register_meta)
validated = param_validator.validate(tool_result, register_meta)
if not validated.ok:
    return ask_clarification(validated.error)
data = await onec_client.execute_tool(tool_result)
if not data.success:
    return handle_onec_error(data)
```

### Файловая структура
```
api/
  main.py              # упрощённый flow
```

## Изменения на стороне 1С

### HTTP-сервис: POST `/analytics/execute`

### Структура модуля
```
МодульHTTPСервиса
├── ОбработатьЗапрос()           — роутер по tool
├── ВыполнитьAggregate()         — ВЫБРАТЬ СУММА(...) ГДЕ ...
├── ВыполнитьGroupBy()           — + СГРУППИРОВАТЬ ПО
├── ВыполнитьTopN()              — + УПОРЯДОЧИТЬ ПО ... ПЕРВЫЕ N
├── ВыполнитьTimeSeries()        — + ГОД(), МЕСЯЦ() в SELECT
├── ВыполнитьCompare()           — два значения одного измерения + diff/percent
├── ВыполнитьRatio()             — два запроса по разным показателям + деление
├── ВыполнитьFiltered()          — ИМЕЮЩИЕ СУММА(...) > &Порог
├── ПостроитьУсловияОтбора()     — общая: filters + period → текст ГДЕ
├── ВалидироватьПараметры()      — проверка values vs метаданные
└── СформироватьОтвет()          — результат → JSON
```

### Ключевые моменты

**`ПостроитьУсловияОтбора()`** — переиспользуется всеми 7 обработчиками:
- Принимает `filters` + `period`
- Строит `ГДЕ ... И ГОД(Дата) = &Год И МЕСЯЦ(Дата) = &Месяц`
- Подставляет через `Запрос.УстановитьПараметр()`

**`ВыполнитьCompare()`** — один запрос с `ГДЕ Сценарий В (&Значения)` + `СГРУППИРОВАТЬ ПО Сценарий`, вычисляет diff/percent

**`ВыполнитьRatio()`** — один запрос с `ГДЕ Показатель В (&Значения)` + `СГРУППИРОВАТЬ ПО Показатель`, деление с проверкой на ноль

**`ВыполнитьFiltered()`** — использует `ИМЕЮЩИЕ СУММА(Сумма) > &Порог`. Оператор подставляется через switch/case (5 вариантов), не конкатенацией строки — защита от инъекций

**`ВалидироватьПараметры()`** — проверяет register, имена измерений/ресурсов, значения против справочников. Возвращает structured error с `allowed_values`.
