# Multi-value filters + year-only period + регистр-доки

**Дата:** 2026-04-13
**Статус:** черновик на ревью

## Проблема

По итогам первого прогона на реальных вопросах вылезли три дефекта:

1. **Только первое значение фильтра попадает в запрос.**
   «Какая выручка за 2024 у ДЗО-1 и ДЗО-2?» → в params уходит `ДЗО="ДЗО-1"`,
   второе ДЗО молча теряется. Причина: `company` в JSON Schema — `type: string`.
2. **Запрос за весь год не работает.**
   «Выручка за 2024 год» → `month` обязателен в schema. Без месяца срабатывает
   клэрификация; с месяцем модель ставит `month=12` и получаем только декабрь.
3. **Имя регистра в `registers.yaml` записано с префиксом типа
   (`РегистрСведений.Витрина_Дашборда`).** 1С принимает только идентификатор —
   при передаче префикс ломает маршрутизацию обработчика.

Также нужно зафиксировать два организационных момента, без которых новые
регистры добавляются с ошибками:

- При каждом новом измерении надо **вручную добавить Latin-ключ** в
  `_KEY_TO_DIM` / `_dim_key` в `api/tool_defs.py`. Пропущенный маппинг =
  кириллица в JSON Schema = ломаный tool calling.
- Поля `technical` и `default` в `registers.yaml` нужно **ревьювить вручную**
  (или через интервью `sync_metadata.py`). Без этого технические поля
  протекают в schema модели, а отсутствующие defaults приводят к лишним
  уточнениям.

И дополнительно — caveat: сейчас в системе один tool (`query` с `mode` enum),
поэтому даже Qwen3.5:2b не ошибается в выборе функции. Это **не означает
робастность tool calling** — при появлении 2+ tools обязателен повторный
калибровочный прогон.

## Решение

### 1. Фильтры — всегда массив (B2)

Все фильтр-измерения в JSON Schema превращаются в `type: "array"` с
`items.enum`. Небольшие модели часто шлют строку — `_normalize_params`
толерантно оборачивает строку в одноэлементный массив. В 1С уходит
всегда массив → BSL строит `<Имя> В (&Значения)` без ветвления.

### 2. Период — месяц опционален (A1)

`month` становится optional. Если `null` / отсутствует → условие по
периоду сводится к `ГОД(<date_dim>) = &Год`. Модель учим через few-shot:
«за 2024 год» ⇒ `year=2024` без `month`.

Кварталы/полугодия **out of scope** — добавятся, когда появится реальный
пользовательский кейс.

### 3. Имя регистра — без префикса типа

- `registers.example.yaml`: `name: Витрина_Дашборда` вместо
  `РегистрСведений.Витрина_Дашборда`.
- Тип хранится отдельно (`type: information_register` уже есть).
- Спецификация `/analytics_execute` и `docs/1c-http-service-module.md`
  обновляются: `register` в payload — идентификатор, тип приходит
  отдельным полем или выводится в 1С из метаданных.

### 4. Доки для новых регистров

Добавляем раздел в `CLAUDE.md` — чек-лист «Adding a new register»:

1. Имя в `registers.yaml` — только идентификатор.
2. Для каждого измерения — Latin-ключ в `_KEY_TO_DIM` и `_dim_key`.
3. Ручная ревизия `technical` и `default` (или интервью sync_metadata).
4. `python3 scripts/seed_metadata.py` → проверить `.env` MODEL_NAME перед
   калибровкой.

Плюс caveat про «один tool ≠ робастный tool calling».

## Контракты

### JSON Schema (api/tool_defs.py)

```python
# Фильтр-измерение (было — type: string):
"company": {
    "type": "array",
    "items": {"type": "string", "enum": ["Консолидация", "ДЗО-1", "ДЗО-2"]},
    "description": (
        "Company values. Pass as array even for a single value. "
        "For 'ДЗО-1 и ДЗО-2' pass both: [\"ДЗО-1\", \"ДЗО-2\"]."
    )
}

# Период:
"year": {"type": "integer", "description": "Year, e.g. 2025"},
"month": {"type": "integer", "description": "Month 1-12. Omit for whole-year queries."},

# required: ["mode", "resource", "year"]   # month убран из required
```

### Python → 1С payload (api/onec_client.py)

```json
{
  "register": "Витрина_Дашборда",
  "tool": "aggregate",
  "params": {
    "resource": "Сумма",
    "filters": {
      "Сценарий": ["Факт"],
      "ДЗО": ["ДЗО-1", "ДЗО-2"]
    },
    "period": {"year": 2024}
  }
}
```

Правила нормализации (`api/tool_caller.py::_normalize_params`):

- Если модель прислала строку `"company": "ДЗО-1"` → `filters["ДЗО"] = ["ДЗО-1"]`.
- Пустой массив `"company": []` → отбрасываем из filters (падаем на default, если есть).
- `month` отсутствует/`null` → в `period` кладём только `{"year": Y}`.

### 1С BSL (`ПостроитьУсловияОтбора`)

- Для каждого фильтра: `<Имя> В (&<Имя>)`, параметр — `Новый Массив()` из входных строк.
- Период: `ГОД(<date_dim>) = &Год` всегда; `И МЕСЯЦ(<date_dim>) = &Месяц` только
  при наличии `period.month`.
- Без ветвления «скаляр vs массив» — контракт однороден.

## Валидация (api/param_validator.py)

- `filters[dim]` ожидается как список. Если строка — нормализуется в tool_caller до входа сюда.
- Для каждого элемента списка — существующий `_resolve_enum`. Первая ошибка собирается в `errors`.
- `month` валидируется только если задан (диапазон 1..12).
- `year` остаётся обязательным (2020..2030).

## Error flow

- Неизвестное значение одного из элементов массива → `invalid_params` с указанием индекса:
  `company[1]: copy EXACTLY one of [...]. You wrote "ДЗО-X".`
- Пустой массив + нет default для required dim → `missing_params`, переспрос.
- `year` отсутствует → `missing_params`, переспрос как раньше.

## Тесты

1. `tests/test_tool_defs.py`:
   - фильтр-измерения имеют `type: array`, `items.enum` есть;
   - `month` **не** в `required`.
2. `tests/test_tool_caller.py`:
   - модель шлёт строку `"company": "ДЗО-1"` → `filters["ДЗО"] = ["ДЗО-1"]`;
   - модель шлёт массив `["ДЗО-1","ДЗО-2"]` → остаётся массивом;
   - пустой массив → не попадает в filters;
   - `month` отсутствует → `period = {"year": ...}` без ключа `month`.
3. `tests/test_param_validator.py`:
   - fuzzy-resolve по каждому элементу массива;
   - одна ошибка на невалидный элемент, остальные остаются.
4. `scripts/calibrate_tools.py` — два новых кейса:
   - «Выручка за 2024 год» → `year=2024`, `month` отсутствует.
   - «Выручка за 2024 у ДЗО-1 и ДЗО-2» → `company=["ДЗО-1","ДЗО-2"]`.
5. Ручной прогон `uvicorn` + реальный вопрос в вебе после мерджа — правило
   «CI-зелёный ≠ работает в браузере» из CLAUDE.md.

## Файлы, которые меняются

## Out of scope

- Кварталы / полугодия (`month_from`/`month_to`) — добавим, когда понадобится.
- Сравнения за разные периоды (compare mode + разные `year`) — отдельная задача.
- Реальная валидация JSON Schema на уровне Ollama (`minItems` и т.п.) — бесполезно,
  Ollama не энфорсит; полагаемся на `param_validator`.

## Риски

- **5B-модель путается между строкой и массивом.** Смягчение: толерантность
  `string → [string]` + явный пример массива в few-shot.
- **`В (&Знач)` для одного элемента** — для 1С это эквивалент `= &Знач`, риска нет.
- **Старые регистры с префиксом `РегистрСведений.`** — seed-скрипт падает с
  understandable error, миграции не требуется (`registers.yaml` перегенерируется
  из шаблона либо правится руками).
